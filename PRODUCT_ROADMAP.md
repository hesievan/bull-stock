## 三、优化路线

### Phase 1: 运维优化（本周）

| # | 任务 | 工作量 | 收益 |
|---|------|--------|------|
| 1.1 | **数据库压缩备份** | 0.5天 | 换电脑时快速恢复 |
| 1.2 | **飞书通知定制** | 1天 | 只推真正重要的信号 |
| 1.3 | **本地一键运行** | 0.5天 | 不依赖CI也能跑 |

**具体方案：**

#### 1.1 数据库压缩备份

```bash
# 一键备份
python scripts/db_compress.py compress
# 输出: data/heat_index.db.gz (742MB)

# 恢复
python scripts/db_compress.py decompress
```

建议：每周手动备份一次，或添加cron自动备份。

#### 1.2 飞书通知定制

修改 `config/default.yaml`：

```yaml
notification:
  # 只在这些条件触发时推送
  notify_on:
    - condition: "score >= 65"      # 红区才推
    - condition: "score >= 55 AND change > 5"  # 橙区且单日涨5分以上
    - condition: "level_change: green->red"     # 绿直接跳红
  
  # 静默时段（不推送）
  quiet_hours:
    start: "22:00"
    end: "08:00"
  
  # 防抖增强
  debounce:
    red_days: 2        # 连续2天红才推
    recovery_days: 1   # 恢复1天就推
```

#### 1.3 本地一键运行

```bash
# 添加 Makefile 或脚本
make daily        # 运行今日计算
make daily DATE=2026-06-15  # 指定日期
make backup       # 备份数据库
make serve        # 启动本地API
```

### Phase 3: 智能增强（2-4周）

| # | 任务 | 工作量 | 收益 |
|---|------|--------|------|
| 3.1 | **热度变化归因** | 3天 | 知道为什么涨/跌 |

| 3.4 | **多时间框架** | 2天 | 周线/月线热度 |

**具体方案：**

#### 3.1 热度变化归因

```python
def explain_change(today, yesterday):
    """解释热度变化原因"""
    changes = []
    for dim in ['valuation', 'fund', 'sentiment', 'technical', 'structure']:
        t = today['dimensions'][dim]['score']
        y = yesterday['dimensions'][dim]['score']
        if abs(t - y) > 3:
            direction = "↑" if t > y else "↓"
            changes.append(f"{dim}: {y:.1f} → {t:.1f} {direction}")
    return changes
```

输出示例：
```
今日热度 46.5 (昨日 44.2, +2.3)
主要变化:
- 情绪: 45.2 → 52.1 ↑ (换手率上升)
- 资金: 58.3 → 55.1 ↓ (北向流出)
```

#### 3.2 异常检测

自动标记：
- 某维度突然跳变 >10分
- 多维度同时 >70（极端情况）
- 历史分位 >95% 的指标

#### 3.3 趋势预测

基于历史模式匹配：
```
当前模式: 估值56 + 情绪49 + 资金59
历史相似: 2020-07 (牛市启动期)
预测: 未来30天热度可能上升至60-65
```

## 六、核心原则

1. **够用就好** — 不做花哨功能，只解决实际痛点
2. **数据安全** — 数据库备份是第一优先级
3. **减少噪音** — 通知只推真正重要的
4. **可解释** — 不仅知道热度，还要知道为什么
