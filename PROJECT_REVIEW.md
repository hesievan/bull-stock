| 项目 | 详情 |
|------|------|

## 二、架构评估

### 2.1 目录结构: ⭐⭐⭐⭐ (良好)

```
src/                    # 核心代码 (清晰分层)
  ├── config.py         # 配置加载 (15行, 简洁)
  ├── data/
  │   ├── database.py   # SQLite管理 (242行)
  │   └── fetcher.py    # 数据获取 (451行)
  ├── indicators/
  │   └── calculator.py # 指标计算 (1360行) ← 核心引擎
  └── output/
      └── json_writer.py # 输出+飞书通知 (289行)

scripts/                # 辅助脚本 (30+个, 用于回填/回测/分析)
web/                    # 前端SPA
config/                 # YAML配置
reports/                # 日报/回测报告
data/                   # SQLite数据库
```

**优点**:
- 核心模块职责单一，分层清晰
- src/ 放业务逻辑，scripts/ 放一次性/辅助脚本
- 配置外置(YAML)，便于调整

**问题**:
- `calculator.py` 过大(1360行)，承担了主指数+板块热度两个职责
- `scripts/` 目录有30+脚本，部分功能重叠(多个backfill脚本)
- 无 `__init__.py` 包导出声明（虽然能运行）

### 2.2 依赖管理: ⭐⭐⭐ (基本可用，有缺陷)

```
tushare>=1.4.0
akshare>=1.15.0
pandas>=2.0.0
numpy>=1.24.0
```

**问题**:
- 缺少 `pyproject.toml` 或 `setup.py`，无法作为Python包安装
- 缺少 `requirements-dev.txt` (测试/lint工具)
- `yaml` 依赖未声明 (config.py 用到 `yaml.safe_load`)
- 未锁定版本号，存在破坏性更新风险
- GitHub Actions 中安装的包与 requirements.txt 不完全一致 (`pip install akshare pandas numpy` 缺少 tushare/yaml)

## 三、代码质量评估

### 3.1 核心计算引擎 (calculator.py): ⭐⭐⭐⭐

**优点**:
- 16个子指标实现完整，覆盖6个维度
- 每个指标独立方法，命名清晰
- 使用预计算表(index_pe/daily_turnover等)优化查询性能
- 3σ异常值过滤 + 动态权重归一化
- 历史分位计算逻辑正确

**问题**:
| 问题 | 严重度 | 位置 |
|------|--------|------|
| `_conn()` 连接未显式关闭 | 中 | calculator.py:94-99 |
| `_load_hist_constituents` 无缓存失效机制 | 低 | calculator.py:134-174 |
| 部分方法直接 `sqlite3.connect()` 绕过 `get_conn()` 上下文管理器 | 中 | calculator.py:98, 1262 |
| `calculate_sector_heat` 未使用 `get_conn()` | 中 | calculator.py:1262 |
| 硬编码阈值(如PE≤500, PB≤10)缺乏配置化 | 低 | calculator.py:204, 243 |

### 3.2 数据获取 (fetcher.py): ⭐⭐⭐

**优点**:
- tushare API 封装良好，带重试和限速
- 增量更新逻辑完善
- 错误处理覆盖每个API调用

**问题**:
| 问题 | 严重度 | 位置 |
|------|--------|------|
| `TUSHARE_TOKEN` 从环境变量读取，但 run_daily.py 从 .env 文件读取 — 逻辑不一致 | 中 | fetcher.py:24, run_daily.py:19-24 |
| `fetch_daily_basic_to_stock_daily` 直接用 `sqlite3.connect()` 而非 `get_conn()` | 中 | fetcher.py:325 |
| `_save` 函数重复包装 `save_dataframe` | 低 | fetcher.py:46-48 |
| `fetch_m2_history` 忽略 start/end 参数 | 低 | fetcher.py:413 |

### 3.3 数据库层 (database.py): ⭐⭐⭐⭐

**优点**:
- 上下文管理器模式(`get_conn`)正确实现事务管理
- WAL模式 + NORMAL同步，性能合理
- UPSERT逻辑(`INSERT OR REPLACE`)处理增量写入
- 表结构设计合理，字段注释清晰

**问题**:
| 问题 | 严重度 | 位置 |
|------|--------|------|
| `save_dataframe` 使用临时表 `_tmp_upsert` — 高并发下可能冲突 | 低 | database.py:202-205 |
| `SCHEMA` 中部分表不在主流程使用(stock_balance, sector_heat等) | 低 | database.py:76-168 |
| 无数据库版本迁移机制 | 中 | - |

### 3.4 输出模块 (json_writer.py): ⭐⭐⭐⭐

**优点**:
- 防抖逻辑设计合理(连续2天才触发红区通知)
- 飞书通知格式美观，信息密度高
- 历史数据去重追加，无重复

**问题**:
| 问题 | 严重度 | 位置 |
|------|--------|------|
| 飞书 Webhook URL 硬编码在代码中(虽然有环境变量覆盖) | 中 | json_writer.py:19-21 |
| `analyze_state` 的 `consecutive` 计算逻辑在 `current_level != prev_day level` 时可能不准确 | 低 | json_writer.py:131-139 |

## 五、工程化评估

### 5.1 测试覆盖: ⭐ (严重不足)

- **无任何单元测试** (`test*` 文件未找到)
- 无 pytest/unittest 配置
- 无测试覆盖率报告
- 唯一的"测试"是 `scripts/verify_all_fixes.py` 和 `scripts/verify_p0_fix.py`，属于手动验证脚本

**风险**: 指标计算逻辑无自动化回归测试，任何改动可能引入静默错误。

### 5.2 CI/CD: ⭐⭐⭐ (基本可用)

GitHub Actions 配置:
```yaml
schedule: '30 8 * * 1-5'  # UTC 08:30 = 北京 16:30
```

**优点**:
- 自动定时运行
- 支持手动触发
- 自动提交结果到仓库

**问题**:
| 问题 | 严重度 |
|------|--------|
| 无测试步骤 — 部署前不验证代码正确性 | 高 |
| 无 lint/typecheck 步骤 | 中 |
| 安装依赖时缺少 tushare 和 yaml | 高 |
| 无错误告警机制(除了飞书) | 低 |
| `git push` 无认证配置(可能失败) | 高 |

### 5.3 日志与监控: ⭐⭐⭐

- 有结构化日志(logging模块)
- 有 `run_status.json` 记录每步状态
- 有飞书通知(防抖逻辑)
- 缺少: 性能监控、异常告警、数据质量检查

### 5.4 配置管理: ⭐⭐⭐

- YAML配置文件，支持环境变量覆盖
- 热度阈值可配置
- 缺少: 多环境配置(dev/prod)、敏感信息管理

## 十、总结

这是一个**功能完善、文档详尽的量化分析项目**。核心指标计算逻辑设计合理，数据获取和存储架构清晰，飞书通知和前端展示体验良好。

**主要优势**:
- 6维度16指标体系设计科学，回测验证通过
- 增量更新机制完善，2772天回测0失败
- 文档质量高于同类项目

**主要短板**:
- 零测试覆盖 — 这是最大风险
- CI/CD配置有缺陷(依赖缺失、无测试步骤)
- 代码有少量工程化问题(连接管理不一致、配置不统一)

**建议**: 优先补充单元测试和修复CI，这两个改动工作量小但对项目长期健康度影响最大。
